> So, when witnessing a runtime error in JavaScript that reads something like this: site.js:74:93. It means; line 74, character 93. Not line 74 : line 93. Think of it like; line : column. So, 74:93 means; "On line 74, the 93rd character." Based on my current experience with JavaScript, this tends to mean look of the runtime error on line 74, at it's 93rd character..... but more to the point here -- the 93rd character in this instance represented the last character of line 74.... and line 74 represented the first line of code in a block of programming that was utilizing DOM (document object model) manipulation, whereby it would do as follows; document.getElementById("monthlyPaymentResult").innerText = `$${data.monthlyPayment.toFixed(2)}`; and I'm pretty sure that semicolon on that line was the last and 93rd character of it. It was trying to retrieve data; document.getElementById("monthlyParymentResult") -- this an example of how JavaScript attempts to find something.... and then the .innertext = `$${data.monthlyPayment.toFixed(2)}`; is what JavaScript can do to convert that data it's found.... but the reason for why the error showed up in the first place is likely because when JavaScript was attempting to retrieve data here, it was given something that was undefined.... and this could have been due to the way I structured the UI, via html and JavaScript, by way of it requiring a valid input.... whereas if it's left blank, it gets treated as undefined, even if the placeholder is defined as 0 in html.

> Often times, I'll get this ambiguous error at runtime in JavaScript called, 'Error 400' Bad Request. In a broad sense, this error can mean a number of different things, and may show up for a number of different reasons, but it's to realize that getting such an error doesn't necessarily mean that your code is broken. It's more likely due to a sever receiving data that it doesn't 'like' or accept.... which again can be for a number of different reasons.... and because of my first final project (the finance one), which is full-stack based.... it's likely due to JavaScript receiving one or more inputs on the user interface that it classified as undefined or as NaNs.... which is the first major issue that this error doesn't necessarily disclose, and then by the time this undefined input gets sent back to my controller in C#, C# doesn't know what to do with it, because JavaScript essentially didn't know what to do with it either, there fore, the server returns an Error 400 'Bad Request'. So, check your inputs before frontend execution..... check JavaScript syntax as well as html syntax, and then C# syntax as a last resort, but in a worst case scenario regarding this sort of instance, it may not be a bad idea to restart both the frontend and backend local host servers.
